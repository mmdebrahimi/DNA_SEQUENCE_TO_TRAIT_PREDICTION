# Lessons Learned

Concise practical lessons from building Phase 1. Append-only.

## 2026-05-11 / 2026-05-12

- **Inter-wave `/brainstorm` quality gates catch real contract-violation bugs.** Three brainstorm rounds (post-Wave-1, post-Wave-2, post-Wave-3) each surfaced 3 grounded issues that would have compounded into Wave N+1. Pattern repeats: `/execute-plan` agents fill plan interfaces but under-specify integration boundaries. Run `/brainstorm` between every wave; don't skip "to save time."
- **Plan-described interfaces leave integration boundaries under-specified.** Wave 1: pilot `NotImplementedError` + refseq ZIP-not-unpacked. Wave 2: BV-BRC strain_id ≠ NCBI accession + save/load asymmetry + 2.25M HDF5 file-opens. Wave 3: calibration CV uses majority class + attribution uses locus_tag never matches catalog. None of these were ambiguous in the spec — they were just under-specified at the boundary where two modules meet.
- **"Defer hardening" decisions reverse fast when the next wave starts integrating.** Wave 2.5 was initially deferred ("accept the debt"); user reversed to "apply now" before Step 18 ran on top. Lesson: when a contract gap is identified, fix it BEFORE the dependent layer builds on it — the cost of fixing later is higher than fixing now.
- **`/probe` adds value AFTER `/brainstorm` when ChatGPT framings need second-opinion calibration.** ChatGPT's "AlphaFold for DNA" + "RL for sequence learning" framings were directionally interesting but partly misleading; `/probe` round caught the bad parts (RL is supervised; AlphaFold is non-interpretable; phenotype ≠ generated text) and kept the good parts (Tier 1-5 success rubric; attribution refinement pipeline; multimodal long-term framing).
- **Always pass user verbatim values, even when factually wrong.** The original DNA goal contained "3-5 proteins" (factually wrong — DNA is 4 nucleotides). Stripping the error from the `/idea-anchor` rephrase silently hid it from `/project-init`'s Empirical Concerns sub-gate; surfacing it explicitly under Assumptions + Blunt Opinion is the right shape. Plan-contract clarification later: `/idea-anchor` Step 1 now emits `Verbatim Input` + `Formal Rephrase` separately.
- **`/execute-plan` parallel mode (`claude -p --worktree`) was untested in cross-session orchestration.** User chose sequential implementation in-conversation instead. Worked cleanly, no subprocess auth/stdout fragility. Default to sequential when the user is actively engaged + token cost is acceptable.
- **`.gitignore`'s `data/` pattern matched both top-level `data/` AND `dna_decode/data/` subpackage.** Anchored `/data/` (leading slash = repo root only) fixed it. General lesson: bare directory names in `.gitignore` are recursive; anchor with leading slash when you mean "top-level only."
- **Solo / part-time + Python-ML strong + genomics-new is a real user profile that changes recommendations.** Codex's review explicitly factored this: rejected curriculum training, AlphaFold-style pair representation, multi-task shared encoder for Phase 1 — all "right architecture" for a team with compute budget, wrong for a solo Phase 1 ship.
- **`/save-plan` after `/review` produces a useful ship-path-delta artifact distinct from the original technical plan.** The synthesis verdict (HOLD + selective contraction) becomes a separate plan file that captures *what changes* without re-stating the technical plan. Future contributors read both.
- **Annotation gene_id semantics vary by source.** GFF3 `ID=` vs `Name=` vs `gene=` attributes get collapsed into one `gene_id` column in `parse_gff3`. Bakta emits `ID=g1` + `locus_tag=TAG_001`; RefSeq emits `gene=gyrA`. Downstream code matching against a gene-symbol catalog (`tier_classify`) needs to try BOTH `gene_id` + `locus_tag` to handle annotation-source variance.

## 2026-05-13

- **`AutoModel` ↔ `AutoModelForMaskedLM` equivalence is NOT a safe assumption under `trust_remote_code=True`.** Author-controlled modeling code can define architecturally distinct variants (different layer shapes), not just head-differential ones. Verified empirically against `InstaDeepAI/nucleotide-transformer-v2-100m-multi-species`: `AutoModel.from_pretrained` fails at load time with a `state_dict` shape mismatch (checkpoint has `Linear[4096, 512]`; AutoModel code defines `Linear[512, 2048]`). The "drop the MLM head, use the base" refactor pattern that works for standard HuggingFace BERT-family models does NOT generalize across all `trust_remote_code` models. Lesson: gate any AutoModel-swap refactor on an empirical load + numerical-equivalence test BEFORE shipping; do not infer compatibility from sibling-model patterns.
- **A baseline that doesn't produce meaningful variance on the actual cohort isn't a baseline.** The 12-strain cipro mini-cohort has 12 unique MLST strings (one per strain). Replacing the `hash(mlst) % 10` placeholder with a real MLST parser maps every strain to a singleton clade ID → under LOSO, every held-out strain is in a clade unseen during training → `predict_clade_only` falls back to `global_positive_rate` → clade-only AUROC ≈ 0.5 regardless of foundation model quality. The "fix" makes the baseline numerically worse (deterministic noise) than the placeholder (synthetic groups of size >1). Lesson: validate that a baseline produces non-degenerate output on the specific cohort BEFORE using it as a comparator in an acceptance gate. A textbook-correct baseline can be empirically useless at small N when the grouping variable is high-cardinality.
- **External literature synthesis (via `/research`) can reveal first-principles project-design mismatches that empirical iteration would discover much later.** Today's `/research` on SOTA AMR classifier heads surfaced that GBDT performance is positively correlated with samples-to-features ratio (McElfresh NeurIPS 2024 Table 16, Section D.4); neural nets show the opposite. dna_decode's setup (N=12-150 strains × 512-dim NT embeddings, samples ≪ features) is in the GBDT-disfavored regime — the OPPOSITE of where XGBoost was designed to win. This is a first-principles design concern, not an empirical-tuning issue. Lesson: at project-direction decision points (architecture choices, classifier-head selection, model-class commitments), do a literature pass BEFORE committing weeks of empirical work; the literature can identify regime mismatches that take many empirical cycles to discover.
- **Cross-engine convergence (`/brainstorm` Codex + ChatGPT independently) is a strong project-state signal.** Today's B-resolution decision was independently picked as B-B by both engines on essentially the same evidence; both flagged Python `hash()` as a reproducibility bug; both narrowed D2's ARCHITECTURE.md scope to per-line judgment. When two independent reviewers converge on a non-obvious decision (especially one against the "fuller scope is better" prior), treat the decision as higher-confidence than either engine alone. Equally important corollary: cross-engine *divergence* on a specific detail (e.g., ChatGPT preferred `zlib.crc32`, Codex used `hashlib.blake2b`) is low-signal noise — pick the simpler option and move on. The convergence-vs-divergence distinction matters more than the count of engines consulted.
- **Project docs claimed RTX 4090 throughout Phase 1; real hardware is GTX 860M (4 GiB Maxwell, 2014).** Discovered 2026-05-13 during first real-embedding run when NT populate OOM'd at batch=32. Practical impact: DNABERT-2 + 4-bit Evo unavailable on this GPU; NT v2 100M works but slowly (~9 min/strain on 12-strain cohort). Lesson: validate hardware-claimed-in-docs via `torch.cuda.get_device_properties()` early in any new project setup, not retroactively when scaling fails. The plan ledger should record the actual `device_name + total_memory + compute_capability` measured at toolchain-bootstrap time, not the aspirational target hardware.
