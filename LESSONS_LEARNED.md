# Lessons Learned

Concise practical lessons from building Phase 1. Append-only.

## 2026-05-11 / 2026-05-12

- **Inter-wave `/brainstorm` quality gates catch real contract-violation bugs.** Three brainstorm rounds (post-Wave-1, post-Wave-2, post-Wave-3) each surfaced 3 grounded issues that would have compounded into Wave N+1. Pattern repeats: `/execute-plan` agents fill plan interfaces but under-specify integration boundaries. Run `/brainstorm` between every wave; don't skip "to save time."
- **Plan-described interfaces leave integration boundaries under-specified.** Wave 1: pilot `NotImplementedError` + refseq ZIP-not-unpacked. Wave 2: BV-BRC strain_id â‰  NCBI accession + save/load asymmetry + 2.25M HDF5 file-opens. Wave 3: calibration CV uses majority class + attribution uses locus_tag never matches catalog. None of these were ambiguous in the spec â€” they were just under-specified at the boundary where two modules meet.
- **"Defer hardening" decisions reverse fast when the next wave starts integrating.** Wave 2.5 was initially deferred ("accept the debt"); user reversed to "apply now" before Step 18 ran on top. Lesson: when a contract gap is identified, fix it BEFORE the dependent layer builds on it â€” the cost of fixing later is higher than fixing now.
- **`/probe` adds value AFTER `/brainstorm` when ChatGPT framings need second-opinion calibration.** ChatGPT's "AlphaFold for DNA" + "RL for sequence learning" framings were directionally interesting but partly misleading; `/probe` round caught the bad parts (RL is supervised; AlphaFold is non-interpretable; phenotype â‰  generated text) and kept the good parts (Tier 1-5 success rubric; attribution refinement pipeline; multimodal long-term framing).
- **Always pass user verbatim values, even when factually wrong.** The original DNA goal contained "3-5 proteins" (factually wrong â€” DNA is 4 nucleotides). Stripping the error from the `/idea-anchor` rephrase silently hid it from `/project-init`'s Empirical Concerns sub-gate; surfacing it explicitly under Assumptions + Blunt Opinion is the right shape. Plan-contract clarification later: `/idea-anchor` Step 1 now emits `Verbatim Input` + `Formal Rephrase` separately.
- **`/execute-plan` parallel mode (`claude -p --worktree`) was untested in cross-session orchestration.** User chose sequential implementation in-conversation instead. Worked cleanly, no subprocess auth/stdout fragility. Default to sequential when the user is actively engaged + token cost is acceptable.
- **`.gitignore`'s `data/` pattern matched both top-level `data/` AND `dna_decode/data/` subpackage.** Anchored `/data/` (leading slash = repo root only) fixed it. General lesson: bare directory names in `.gitignore` are recursive; anchor with leading slash when you mean "top-level only."
- **Solo / part-time + Python-ML strong + genomics-new is a real user profile that changes recommendations.** Codex's review explicitly factored this: rejected curriculum training, AlphaFold-style pair representation, multi-task shared encoder for Phase 1 â€” all "right architecture" for a team with compute budget, wrong for a solo Phase 1 ship.
- **`/save-plan` after `/review` produces a useful ship-path-delta artifact distinct from the original technical plan.** The synthesis verdict (HOLD + selective contraction) becomes a separate plan file that captures *what changes* without re-stating the technical plan. Future contributors read both.
- **Annotation gene_id semantics vary by source.** GFF3 `ID=` vs `Name=` vs `gene=` attributes get collapsed into one `gene_id` column in `parse_gff3`. Bakta emits `ID=g1` + `locus_tag=TAG_001`; RefSeq emits `gene=gyrA`. Downstream code matching against a gene-symbol catalog (`tier_classify`) needs to try BOTH `gene_id` + `locus_tag` to handle annotation-source variance.

## 2026-05-13

- **`AutoModel` â†” `AutoModelForMaskedLM` equivalence is NOT a safe assumption under `trust_remote_code=True`.** Author-controlled modeling code can define architecturally distinct variants (different layer shapes), not just head-differential ones. Verified empirically against `InstaDeepAI/nucleotide-transformer-v2-100m-multi-species`: `AutoModel.from_pretrained` fails at load time with a `state_dict` shape mismatch (checkpoint has `Linear[4096, 512]`; AutoModel code defines `Linear[512, 2048]`). The "drop the MLM head, use the base" refactor pattern that works for standard HuggingFace BERT-family models does NOT generalize across all `trust_remote_code` models. Lesson: gate any AutoModel-swap refactor on an empirical load + numerical-equivalence test BEFORE shipping; do not infer compatibility from sibling-model patterns.
- **A baseline that doesn't produce meaningful variance on the actual cohort isn't a baseline.** The 12-strain cipro mini-cohort has 12 unique MLST strings (one per strain). Replacing the `hash(mlst) % 10` placeholder with a real MLST parser maps every strain to a singleton clade ID â†’ under LOSO, every held-out strain is in a clade unseen during training â†’ `predict_clade_only` falls back to `global_positive_rate` â†’ clade-only AUROC â‰ˆ 0.5 regardless of foundation model quality. The "fix" makes the baseline numerically worse (deterministic noise) than the placeholder (synthetic groups of size >1). Lesson: validate that a baseline produces non-degenerate output on the specific cohort BEFORE using it as a comparator in an acceptance gate. A textbook-correct baseline can be empirically useless at small N when the grouping variable is high-cardinality.
- **External literature synthesis (via `/research`) can reveal first-principles project-design mismatches that empirical iteration would discover much later.** Today's `/research` on SOTA AMR classifier heads surfaced that GBDT performance is positively correlated with samples-to-features ratio (McElfresh NeurIPS 2024 Table 16, Section D.4); neural nets show the opposite. dna_decode's setup (N=12-150 strains Ã— 512-dim NT embeddings, samples â‰ª features) is in the GBDT-disfavored regime â€” the OPPOSITE of where XGBoost was designed to win. This is a first-principles design concern, not an empirical-tuning issue. Lesson: at project-direction decision points (architecture choices, classifier-head selection, model-class commitments), do a literature pass BEFORE committing weeks of empirical work; the literature can identify regime mismatches that take many empirical cycles to discover.
- **Cross-engine convergence (`/brainstorm` Codex + ChatGPT independently) is a strong project-state signal.** Today's B-resolution decision was independently picked as B-B by both engines on essentially the same evidence; both flagged Python `hash()` as a reproducibility bug; both narrowed D2's ARCHITECTURE.md scope to per-line judgment. When two independent reviewers converge on a non-obvious decision (especially one against the "fuller scope is better" prior), treat the decision as higher-confidence than either engine alone. Equally important corollary: cross-engine *divergence* on a specific detail (e.g., ChatGPT preferred `zlib.crc32`, Codex used `hashlib.blake2b`) is low-signal noise â€” pick the simpler option and move on. The convergence-vs-divergence distinction matters more than the count of engines consulted.
- **Project docs claimed RTX 4090 throughout Phase 1; real hardware is GTX 860M (4 GiB Maxwell, 2014).** Discovered 2026-05-13 during first real-embedding run when NT populate OOM'd at batch=32. Practical impact: DNABERT-2 + 4-bit Evo unavailable on this GPU; NT v2 100M works but slowly (~9 min/strain on 12-strain cohort). Lesson: validate hardware-claimed-in-docs via `torch.cuda.get_device_properties()` early in any new project setup, not retroactively when scaling fails. The plan ledger should record the actual `device_name + total_memory + compute_capability` measured at toolchain-bootstrap time, not the aspirational target hardware.

## 2026-05-14

- **Anti-predictive output (AUROC=0.000 with symmetric scores around 0.5) signals a bug, not "model broken."** Smoke-gate NT-XGBoost returned `{0.463, 0.537}` as the only two probability values across 12 LOSO folds â€” perfectly symmetric and rank-inverted. Without investigation this looks like "NT failed on real data" and would have triggered an unjustified track demotion. Real cause: `CalibratedClassifierCV` with isotonic regression at N=11 training collapses to two-value symmetric output that inverts the rank order. Fix: `calibrate=False` recovered AUROC=0.750. Lesson: AUROCâ‰ˆ0.000 with two-value symmetric scores = "stop and check calibration / label-column / row-order / class-mapping" BEFORE drawing model-quality conclusions. Genuine model failure produces noisy random-ish scores, not perfectly-symmetric two-value collapse.
- **When ranking next-actions by "highest info per minute," weight by "blocks vs informs."** Three engines converged on three top priorities for Stage 1 (fix gene-presence bug + scale to N=50 + plot PCA/UMAP). Initial intuition picked the cheapest action first (plot, 5 min). Post-/brainstorm correction: the bug-fix is BOTH high-info AND blocking for any downstream comparator matrix; the diagnostic plot is informative but never gates anything. Lesson: among candidate next-actions, separate "blocking" from "diagnostic" â€” blockers must come first even when they're more expensive than the cheapest informative action. "Highest info per minute" is the wrong objective when one action removes a hard dependency.
- **2D embedding visualizations (PCA/UMAP) at small N are sanity checks, not go/no-go gates.** At N=12 with 6R/6S, "clear separation" can come from chance / batch effect / lineage / single outlier axis; "no separation" can come from projection-axis mismatch even when real signal exists in higher dimensions. Plot is informative for interpretation (e.g., does the separation track resistance or lineage?), never decisive for "should we spend the next 6 hours populating embeddings at N=40?" Lesson: don't use 2D-projection visual gestalt to gate spending on the next experimental scale. Use the plot to overlay metadata (phylogroup, source, batch) and check confounders, not to decide whether the project is worth continuing.
- **GFF3 `ID=` attribute is strain-unique by construction; do NOT use as gene-family vocabulary for cross-strain presence/absence features.** Surfaced 2026-05-14. `parse_gff3` was extracting `gene_id` from `attr_map.get("ID", attr_map.get("Name", ""))`, so the gene-presence vocabulary was dominated by strain-unique IDs like `gene-b0001`. Cross-strain vocab overlap was 0.00% under LOSO â†’ test rows all zeros â†’ XGBoost predicted the training class prior â†’ on a balanced 6R/6S cohort the prior always inverts against the held-out label â†’ perfect rank inversion AUROC=0.000. Fix: added a `gene_symbol` column to `AnnotationTable` populated from the GFF3 `gene=` attribute, and downstream gene-family code prefers it. Corollary: when a presence/absence feature pattern returns AUROC=0.000 with all-zero test rows, the upstream vocabulary is strain-unique, not the classifier. Check vocab-overlap percentage BEFORE blaming the model.
- **Defense-in-depth: gate degenerate-feature regimes with an explicit INDETERMINATE verdict.** Same incident. Even after the `gene_symbol` fix, RefSeq GFF3 only carries `gene=` for ~11% of CDSs, so post-fix vocab overlap was 10.84% â€” still degenerate. Rather than reporting a misleading AUROC, the smoke runner now thresholds on median per-fold vocab overlap and returns `INDETERMINATE_IDENTIFIER_OOV` when below 20%. Lesson: when a metric is computable but structurally meaningless under specific data conditions, the right output is a typed verdict, not a number that looks comparable to results from other regimes.
- **Verdict semantics vs budget decisions belong in separate fields.** Critical /brainstorm Round 1 finding on the Stage 1 runner refactor (2026-05-14). Original design folded both "did the gap clear 3pp?" (verdict) and "what should we do next given verdict + CI + fusion behavior?" (budget action) into a single labeled output. Risk: under marginal noisy-PASS results with a CI that crosses zero, the impulse to relabel the verdict to FAIL "to be safe" silently smuggles a budget concern into the scientific verdict â€” motivated reasoning. Fix: split into two fields. `verdict_bucket` = pure function of point gap (CLEAN PASS â‰¥5 pp / NOISY PASS 3-5 pp / FAIL <3 pp), frozen pre-result. `stage2_action` = decision-layer output (`BURST_STAGE_2` / `HOLD_STAGE_2_CI_DEGENERATE` / `ALTERNATIVE_POOLING_RERUN` / `PIVOT_TO_BAKTA`) computed from (verdict, ci_lo, fusion behavior) per a pre-committed mapping table. Verdict stays scientific; action absorbs budget judgment. Pinned by 4 mapping tests + the Stage 1 plan's Verdict-Time Pre-Commitments section. General lesson: when one output is tempted to "swing both ways" under marginal results, the field is doing two jobs â€” split it before the marginal result arrives, not during the post-hoc relabel.
- **Order-explicit alignment contract prevents silent fold-order drift across shared runners.** Stage 1 refactor factored the k-mer + fusion LOSO into `dna_decode/eval/loso_kmer.py` for reuse between the smoke gate and Stage 1 runner. Initial design risk: each runner sorted `strain_ids` internally, so caller-supplied order was discarded and downstream paired comparisons could silently misalign if upstream subsets/orders diverged. Fix: caller passes `strain_ids: list[str]` and the runner respects it verbatim (no sort/filter); `CVResult.strain_ids` property exposes the held-out IDs in fold order; `compute_gate_outcome` raises on NT-vs-k-mer mismatch (gate-bearing) but suppresses the fusion-outperforms note on fusion mismatch (diagnostic-only). Lesson: when two LOSO runners must be paired-comparable, the alignment contract belongs in the type signature (caller-supplied ordered IDs), not the runner internals. Internal sorting feels safe and is actively dangerous.
- **BV-BRC cohort bottleneck is `assembly_accession`, not MLST.** Stage 2 N=150 cohort-build surfaced this 2026-05-14 PM. Initial run produced only 49 cipro-R strains (vs target 75) and the immediate impulse was to relax MLST or assembly-quality thresholds. Layer-by-layer diagnostic (`scripts/diagnose_bvbrc_mlst_gaps.py`) revealed: raw `BVBRC_genome.csv` has 96% MLST coverage (81,961 of 85,114 rows); parser drops 35,790 rows lacking `assembly_accession`; of cipro AST's 568 R total, 491 lack accession and only 77 retain it. With label-stratified MLST-balanced selection, 72 of 72 quality-passing R strains were captured. Lesson: when a cohort comes up short by class, profile the join layers BEFORE relaxing filters. The bottleneck is rarely where the symptom appears â€” often a sibling field that gates downstream feasibility (here: can-we-download-the-genome).
- **`build_cohort`'s diversity-only selection doesn't stratify by binary label; per-class selection matters at cohort ceilings.** Stage 2 N=150 cohort with default `build_cohort`: 49R / 101S. Same candidate pool with per-class `_mlst_balanced_selection` (R pool + S pool selected independently): 72R / 75S (within slack of 75/75 target). The default algorithm picks N strains by MLST diversity across the full pool â€” which oversamples the abundant class when there are more S-options-per-MLST than R-options-per-MLST. Lesson: for binary-classification cohorts under target-N constraints, run `_mlst_balanced_selection` once per class and union, not once over the merged pool. The fix is mechanical and the gain is large at small ceilings.
- **HDF5 EOA-truncation from external-drive USB hiccup is unrecoverable.** Stage 1 N=40 NT populate crashed at ~8 hr wallclock with `errno=13 Permission denied` traceback flooding stderr (2026-05-14 ~16:03). On-disk file size: 132 MB. HDF5 superblock EOA marker: 6,472 bytes. The 132 MB of embeddings were written but never committed via flush â€” superblock points past the end of the populated region, all data orphaned. `h5repack` / `h5dump` can't recover EOA-truncated files. The Seagate Portable D: drive's USB volatility is the recurring failure mode (also lost an earlier populate per `GATE_B_REPORT.md` 2026-05-13). Lesson: any long-running write to external storage MUST flush + checkpoint per logical unit (per strain, not per gene). For HDF5 specifically, call `cache.flush()` after every strain so a USB hiccup costs ~10 min of recompute instead of 8 hours. Better long-term: stage on internal disk, copy to external after completion.
- **Bakta-style GFF3 hides `gene=` on parent rows, not CDS.** Phase A.4 of Stage 2 prep flagged this (2026-05-14 PM). RefSeq-style puts `gene=gyrA` on the CDS row itself; Bakta puts it on the parent `gene` row, with CDS rows linking via `Parent=`. A naive parser that only reads same-row `gene=` would re-enter `INDETERMINATE_IDENTIFIER_OOV` after Bakta re-annotation. Fix: two-pass `parse_gff3` â€” Pass 1 builds `parent_id â†’ gene_symbol` map from gene-type rows; Pass 2 propagates to CDS rows lacking own `gene=`. Same-row `gene=` always wins to preserve RefSeq behavior. Lesson: when integrating a new annotation source, smoke-test parser output on a SINGLE strain BEFORE committing to overnight scale-out. The patch is 10 LOC; discovering the gap mid-scale is hours of wasted compute.

## 2026-05-15

- **Gene-level cache.populate skip + Stage 1 mean-pool = silent half-strain landmine.** `EmbeddingCache.populate(skip_existing=True)` skips at the (strain, gene) dataset path level (`cache.py:296`), not strain level. Stage 1's loader admits any strain with â‰¥1 cached gene + mean-pools whatever is there (`stage1_n40_cipro.py:147-151`). A mid-flush crash leaves a partial gene set that *looks* like a complete strain to Stage 1 â€” the verdict gets built on a half-pool with no signal that anything is wrong. The f9ed79f per-strain flush patch bounds the loss to ONE in-flight strain, doesn't eliminate it. Fix shipped: `EmbeddingCache.verify_complete(expected_genes_by_strain) â†’ CompletenessReport` plus `scripts/probe_nt_cache.py` CLI. Consumer-side discipline: anyone mean-pooling a populated cache MUST call `verify_complete` and bail unless `report.all_complete` is True. 8 regression tests pin the 4 status buckets (complete / partial / absent / corrupt) + the `all_complete=False` rule on partial. Lesson: when a cache stores composite entities (strain = many gene embeddings) and a consumer aggregates whatever is present, the cache MUST expose a completeness primitive â€” silent partial-state is worse than absent-state because consumers can't tell the difference.
- **Git Bash silently breaks Docker volume mounts via MSYS path conversion.** Direct `docker run -v <host>:/db <image> <cmd>` from Git Bash on Windows munges the container path `/db` â†’ `C:/Program Files/Git/db`. Docker daemon doesn't error â€” it creates an unbound ephemeral mount; downloads succeed inside the container, then `--rm` wipes everything. Symptom: command exit 0, host directory empty afterward. Two 40-second AMRFinder DB downloads + a 17-minute Bakta DB download all looked successful but produced 0 bytes on host. Fix: `MSYS_NO_PATHCONV=1 docker run ...` from any Git Bash invocation. Python `subprocess.run` is unaffected (no shell to munge the path), so `tools/docker_runner.py` is safe by construction. Lesson: on Windows hosts, every interactive Docker-volume-mount tutorial assumes a shell that doesn't munge `/<container-path>`. Git Bash does. Either set `MSYS_NO_PATHCONV=1` per-invocation, or route through Python.
- **Bakta `bakta_db download` AMRFinder dep-check fails without conda env activation.** `--entrypoint /opt/conda/bin/bakta_db download ...` skips the image's default bash shell init, leaving the conda env un-activated. `bakta_db`'s AMRFinder dep check (more than `which amrfinder`) then fails despite the binary being on PATH at `/opt/conda/bin/amrfinder`. Symptom: `ERROR: AMRFinderPlus not found or not executable!` on v1.11.4 + v1.12.0 even after image-pull succeeded. Workaround: `--entrypoint /bin/bash -c "bakta_db download ..."` lets conda activate properly. DB extracts cleanly (v6.0 light, 1.34 GB tar.xz â†’ 4.0 GB extracted, MD5 verified via Zenodo DOI 10.5281/zenodo.14916843). Lesson: when invoking a tool inside a conda-based Docker image, bypassing the default entrypoint can break tool-internal dep checks that rely on PATH ordering / conda activation. If a tool errors with "X not found" but `which X` works inside the container, suspect entrypoint-skipping the shell init.
